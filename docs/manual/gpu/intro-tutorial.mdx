本教程向你介绍使用 Mojo 进行 GPU 编程。你将学习如何编写一个在 GPU 上执行向量加法的简单程序，并探索 GPU 编程的基本概念。

完成本教程后，你将：

- 理解基本的 GPU 编程概念，如网格和线程块。
- 学习如何在 CPU 和 GPU 内存之间移动数据。
- 编写和编译简单的 GPU 内核函数。
- 在 GPU 上执行并行计算。
- 理解 GPU 编程的异步特性。

我们将逐步构建所有内容，从基础开始，逐渐增加复杂性。你在这里学到的概念将作为使用 Mojo 进行更高级 GPU 编程的基础。如果你只想看完成的代码，可以在 [GitHub 上获取](https://github.com/modular/modular/tree/main/mojo/examples/gpu-intro)。有关兼容 GPU 和软件要求的列表，请参阅 [GPU 兼容性](/docs/requirements/#gpu-compatibility)。

:::tip

**提示：** 要在 Mojo 中使用 AI 编程助手，请参阅我们的[使用 Mojo AI 技能指南](/docs/tools/skills/)。

:::

## 1. 创建 Mojo 项目

要安装 Mojo，我们推荐使用 [`pixi`](https://pixi.sh/latest/)（其他选项请参阅[安装指南](/install/)）。

    ```sh
    curl -fsSL https://pixi.sh/install.sh | sh
    ```

2. 导航到要创建项目的目录并执行：

    ```bash
    pixi init gpu-intro \
      -c https://conda.modular.com/max/ -c conda-forge \
      && cd gpu-intro
    ```

    这会创建一个名为 `gpu-intro` 的项目目录，添加 Modular conda 包通道，并进入该目录。

    :::tip

    **提示：**
    如果你[将这些通道添加为默认值](/pixi#create-a-project-and-virtual-environment)，可以跳过 `-c` 选项。

    :::

3. 安装 `mojo` 包：

    ```bash
    pixi add mojo
    ```

4. 通过检查项目虚拟环境中安装的 Mojo 版本来验证项目配置正确：

    ```bash
    pixi run mojo --version
    ```

    你应该看到一个版本字符串，指示安装的 Mojo 版本。默认情况下，这应该是最新的每夜版。

    ```bash
    pixi shell
    ```

    稍后，当你想退出虚拟环境时，只需输入 `exit`。

## 2. 获取 GPU 设备引用

[`DeviceContext`](/docs/std/gpu/host/device_context/DeviceContext/) 类型表示 GPU 设备的逻辑实例。它提供了在设备上分配内存、在主机 CPU 和 GPU 之间复制数据以及在设备上编译和运行函数（也称为*内核*）的方法。

使用 [`DeviceContext()`](/docs/std/gpu/host/device_context/DeviceContext/#__init__) 构造函数获取 GPU 设备的引用。如果没有兼容的 GPU 可用，构造函数会抛出错误。你可以使用 [`has_accelerator()`](/docs/std/sys/info/has_accelerator/) 函数检查是否有兼容的 GPU 可用。

让我们先编写一个程序来检查 GPU 是否可用，然后获取 GPU 设备的引用。使用任意编辑器创建一个名为 `vector_addition.mojo` 的文件，包含以下代码：

```mojo title="vector_addition.mojo"
from std.gpu.host import DeviceContext
from std.sys import has_accelerator

def main() raises:
    comptime if not has_accelerator():
        print("未找到兼容的 GPU")
    else:
        ctx = DeviceContext()
        print("找到 GPU:", ctx.name())
```

保存文件并使用 `mojo` CLI 运行：

```bash
mojo vector_addition.mojo
```

你应该看到类似以下的输出（取决于你的 GPU 类型）：

```output
找到 GPU: NVIDIA A10G
```

:::note

Mojo 需要[兼容的 GPU 开发环境](/docs/requirements/#gpu-compatibility)来编译内核函数，否则会抛出编译时错误。在我们的代码中，我们使用 [`comptime if`](/docs/manual/metaprogramming/comptime-evaluation/#comptime-if) 语句在编译时评估 `has_accelerator()` 函数，并只编译 `if` 语句的相应分支。因此，如果你没有兼容的 GPU 开发环境，运行程序时会看到以下消息：

```output
未找到兼容的 GPU
```

在这种情况下，你需要找一个有支持 GPU 的系统来继续本教程。

:::

## 3. 定义简单内核

GPU *内核*只是在 GPU 上运行的函数，在数千或数百万个*线程*上并行执行对大型数据集的特定计算。你可能已经熟悉 CPU 编程中的线程，但 GPU 线程是不同的。在 CPU 上，线程由操作系统管理，可以执行完全独立的任务，如管理用户界面、从数据库获取数据等。但在 GPU 上，线程由 GPU 本身管理。GPU 上的所有线程都执行相同的内核函数，但它们各自处理数据的不同部分。

运行内核时，需要指定要使用的线程数。指定的线程数取决于要处理的数据大小和要实现的并行度。常见的策略是为结果中的每个数据元素使用一个线程。因此，如果你要对两个 1,024 元素的向量进行逐元素加法，你会使用 1,024 个线程。

*网格*是执行内核函数的线程的顶级组织结构。网格由多个*线程块*组成，线程块进一步分为并发执行内核函数的单个线程。GPU 为每个线程块分配唯一的块索引，并为块内的每个线程分配唯一的线程索引。同一线程块内的线程可以通过共享内存共享数据，并使用内置机制同步，但它们不能直接与其他块中的线程通信。在本教程中，我们不会详细介绍为什么或如何这样做，但在编写更复杂的内核时，这是一个需要记住的重要概念。

为了更好地理解网格、线程块和线程是如何组织的，让我们编写一个简单的内核函数来打印线程块和线程索引。将以下代码添加到你的 `vector_addition.mojo` 文件：

```mojo title="vector_addition.mojo"
from std.gpu import block_idx, thread_idx

def print_threads():
    """打印线程 ID。"""

    print(block_idx.x, thread_idx.x, sep="\t")
```

## 4. 编译和运行内核

接下来，我们需要更新 `main()` 函数来为我们的 GPU 编译内核函数，然后运行它，指定网格中的线程块数和每个线程块中的线程数。对于这个初始示例，让我们定义一个由 2 个线程块组成的网格，每个线程块有 64 个线程。

修改 `main()` 函数，使你的程序如下所示：

```mojo title="vector_addition.mojo"
from std.sys import has_accelerator

from std.gpu.host import DeviceContext
from std.gpu import block_idx, thread_idx


def print_threads():
    """打印线程 ID。"""

    print(block_idx.x, thread_idx.x, sep="\t")


def main() raises:
    comptime if not has_accelerator():
        print("未找到兼容的 GPU")
    else:
        ctx = DeviceContext()
        print("block\tthread")
        ctx.enqueue_function[print_threads](
            grid_dim=2, block_dim=64
        )
        ctx.synchronize()
        print("程序结束")
```

保存文件并运行：

```bash
mojo vector_addition.mojo
```

你应该看到类似以下的输出（此处已缩略）。内核将每个线程的块和线程索引打印为制表符分隔的数值，`main()` 函数在启动内核前打印标题行。输出顺序是不确定的，因为线程并发执行。

```output
block	thread
1	32
1	33
1	34
...
0	30
0	31
程序结束
```

:::note[已知限制]

在 Apple silicon GPU 上，GPU 内核内的每个 `print()` 调用目前最多支持一个字符串字面量参数。为确保在所有 GPU 平台上的输出可读性，这里的示例只打印数值，由主机在启动内核前打印列标题。此限制不适用于 NVIDIA 或 AMD GPU。

:::

典型的 CPU-GPU 交互是异步的，允许 GPU 在 CPU 忙于其他工作时处理任务。每个 `DeviceContext` 都有一个关联的操作队列，用于在 GPU 上执行。流中的操作按发出的顺序执行。

[`enqueue_function()`](/docs/std/gpu/host/device_context/DeviceContext/#enqueue_function) 方法编译内核函数并将其排队在给定设备上运行。你必须将内核函数的名称作为编译时 Mojo 参数提供，以及以下参数：

- 内核函数定义指定的任何额外参数（本例中没有）。
- 使用 `grid_dim` 关键字参数指定网格维度。
- 使用 `block_dim` 关键字参数指定线程块维度。

（有关 Mojo 函数参数的更多信息，请参阅 Mojo 手册的[函数](/docs/manual/functions/)部分；有关 Mojo 编译时参数和元编程的更多信息，请参阅[参数](/docs/manual/parameters/)部分。）

:::note[API 变体]

`compile_function_unchecked()` 和 `enqueue_function_unchecked()` 方法已弃用，将在未来版本中移除，因为它们不对内核函数参数执行编译时类型检查。如果传递的值不符合预期的参数类型，你将遇到运行时错误。

`compile_function()` 和 `enqueue_function()` 方法执行编译时类型检查，但只需要一次将内核函数作为参数传入。一旦我们验证这些 API 工作正常，我们将重命名它们为 `compile_function()` 和 `enqueue_function()`，使类型检查成为默认行为。

:::

在此示例中，我们使用 `grid_dim=2` 和 `block_dim=64` 调用编译后的内核函数，这意味着我们使用由 2 个线程块组成的网格，每个线程块有 64 个线程，网格中总共有 128 个线程。

运行内核时，GPU 将网格中的每个线程块分配给一个*流式多处理器*执行。流式多处理器（SM）是 GPU 的基本处理单元，旨在高效执行多个并行工作负载。每个 SM 包含多个核心，这些核心执行在 SM 上运行的线程的实际计算，以及共享资源，如寄存器、共享内存和协调线程执行的控制机制。GPU 上的 SM 数量和核心数量取决于其架构。例如，NVIDIA H100 PCIe 包含 114 个 SM，每个 SM 有 128 个 32 位浮点核心。

此外，当 SM 被分配一个线程块时，它将块划分为多个* warp*，即 32 或 64 个线程的组，具体取决于 GPU 架构。这些线程以*单指令多线程*（SIMT）模型同时执行相同的指令。SM 的 *warp 调度器*协调 warp 在 SM 核心上的执行。

Warp 用于通过最大化吞吐量和最小化控制开销来高效利用 GPU 硬件。由于 GPU 专为高性能并行处理而设计，将线程分组为 warp 可以简化指令调度和执行，降低管理单个线程的复杂性。多个线程块的多个 warp 可以在任何给定时间在 SM 内处于活动状态，使 GPU 能够保持执行单元忙碌。例如，如果某个 warp 的线程被阻塞等待来自内存的数据，warp 调度器可以立即切换执行到另一个准备运行的 warp。

将内核函数入队后，我们要确保 CPU 在退出程序之前等待它完成执行。我们通过调用 `DeviceContext` 对象的 [`synchronize()`](/docs/std/gpu/host/device_context/DeviceContext/#synchronize) 方法来实现这一点，该方法会阻塞直到设备完成其队列中的所有操作。

## 5. 管理网格维度

上一步使用了由 2 个线程块组成的一维网格，每个线程块有 64 个线程。但是，你也可以在二维或三维网格中组织线程块。同样，你可以将线程块中的线程排列在一个、两个或三个维度上。通常，根据要处理的数据的维度确定网格和线程块的维度。例如，你可能会选择一维网格来处理大型向量，二维网格来处理矩阵，三维网格来处理视频帧。

为了更好地理解网格、线程块和线程如何协同工作，让我们修改 `print_threads()` 内核函数来打印每个线程的线程块和线程索引的 `x`、`y` 和 `z` 分量。

```mojo title="vector_addition.mojo"
def print_threads():
    """打印线程 ID。"""

    print(
        block_idx.x,
        block_idx.y,
        block_idx.z,
        thread_idx.x,
        thread_idx.y,
        thread_idx.z,
        sep="\t",
    )
```

然后，更新 `main()` 以打印列标题并将内核函数入队，使用 2x2x1 的线程块网格和每个线程块内 16x4x2 的线程排列：

```mojo title="vector_addition.mojo"
        print("block_idx\t\tthread_idx")
        print("x\ty\tz", "x\ty\tz", sep="\t")
        print("-" * 20, "-" * 20, sep="\t")
        ctx.enqueue_function[print_threads](
            grid_dim=(2, 2, 1),
            block_dim=(16, 4, 2)
        )
```

保存文件并再次运行：

```bash
mojo vector_addition.mojo
```

你应该看到类似以下的输出（此处已缩略）：

```output

x	y	z	x	y	z

1	1	0	0	2	0
1	1	0	1	2	0
1	1	0	2	2	0
...
0	0	0	14	1	0

```

尝试更改网格和线程块维度，看看输出如何变化。

:::note

每个线程块的最大线程数和每个 SM 的线程数是 GPU 特定的。例如，NVIDIA A100 GPU 每个线程块最多有 1,024 个线程，每个 SM 最多有 2,048 个线程。

选择网格和线程块的大小和形状是在最大化可并发执行的线程数和最小化等待从内存加载数据的时间之间的平衡。要处理的数据大小、GPU 上的 SM 数量以及 GPU 的内存带宽等因素都可能影响确定最佳网格和线程块维度。一个通用准则是选择 warp 大小倍数的线程块大小。这有助于最大化 GPU 资源的利用率并最小化管理多个 warp 的开销。

:::

## 6. 为输入向量分配主机内存

在为内核函数创建两个输入向量之前，我们需要了解*主机内存*和*设备内存*之间的区别。主机内存是 CPU 可访问的动态随机存取存储器（DRAM），而设备内存是 GPU 可访问的 DRAM。如果数据在主机内存中，必须先将其显式复制到设备内存，然后才能在内核函数中使用。同样，如果内核函数产生 CPU 稍后要使用的数据，必须将其显式复制回主机内存。

在本教程中，我们将使用 [`HostBuffer`](/docs/std/gpu/host/device_context/HostBuffer/) 类型来表示主机上的向量。`HostBuffer` 是与特定 `DeviceContext` 关联的主机内存块。它支持在主机和设备内存之间传输数据的方法，以及通过索引访问数据元素和打印缓冲区的基本方法集。

让我们更新 `main()` 来为输入向量创建两个 `HostBuffer` 并用值初始化它们。你不再需要 `print_threads()` 内核函数，所以可以删除它以及编译和调用它的代码。

进行这些更改后，你的 `vector_addition.mojo` 文件应该如下所示：

```mojo title="vector_addition.mojo"
from std.gpu.host import DeviceContext
from std.gpu import block_idx, thread_idx
from std.sys import has_accelerator

# 向量数据类型和大小
comptime float_dtype = DType.float32
comptime vector_size = 1000


def main() raises:
    comptime if not has_accelerator():
        print("未找到兼容的 GPU")
    else:
        # 获取附加 GPU 的上下文
        ctx = DeviceContext()

        # 为输入向量创建 HostBuffer
        lhs_host_buffer = ctx.enqueue_create_host_buffer[float_dtype](
            vector_size
        )
        rhs_host_buffer = ctx.enqueue_create_host_buffer[float_dtype](
            vector_size
        )
        ctx.synchronize()

        # 初始化输入向量
        for i in range(vector_size):
            lhs_host_buffer[i] = Float32(i)
            rhs_host_buffer[i] = Float32(i * 0.5)

        print("LHS 缓冲区: ", lhs_host_buffer)
        print("RHS 缓冲区: ", rhs_host_buffer)
```

[`enqueue_create_host_buffer()`](/docs/std/gpu/host/device_context/DeviceContext/#enqueue_create_host_buffer) 方法接受数据类型作为编译时参数，缓冲区大小作为运行时参数，并返回一个 `HostBuffer`。与所有名称以 `enqueue_` 开头的 `DeviceContext` 方法一样，该方法是异步的并立即返回，将操作添加到队列中由 `DeviceContext` 执行。因此，我们需要调用 `synchronize()` 方法来确保操作在使用 `HostBuffer` 对象之前已完成。然后我们可以用值初始化输入向量并打印它们。

现在让我们运行程序来验证目前一切正常。

```bash
mojo vector_addition.mojo
```

你应该看到以下输出：

```output
LHS 缓冲区:  HostBuffer([0.0, 1.0, 2.0, ..., 997.0, 998.0, 999.0])
RHS 缓冲区:  HostBuffer([0.0, 0.5, 1.0, ..., 498.5, 499.0, 499.5])
```

:::note

你可能注意到我们没有显式调用任何方法来释放 `HostBuffer` 分配的主机内存。这是因为 `HostBuffer` 受 Mojo 标准所有权和生命周期机制的约束。Mojo 编译器分析我们的程序以确定对象的所有者或引用最后被使用的点，并自动添加对对象析构函数的调用。在我们的程序中，我们最后在程序 `main()` 方法末尾引用缓冲区。但是，在更复杂的程序中，如果在程序的后续点引用 `HostBuffer`，它可以跨多个内核函数调用持久存在。有关 Mojo 值所有权和值生命周期管理的更多信息，请参阅 Mojo 手册的[所有权](/docs/manual/values/ownership/)和[值生命周期简介](/docs/manual/lifecycle/)部分。

:::

## 7. 将输入向量复制到 GPU 内存并分配输出向量

现在我们已经在 CPU 上分配并初始化了输入向量，让我们将它们复制到 GPU，以便内核函数可以使用它们。同时，我们还将在 GPU 上为保存内核函数结果的输出向量分配内存。

在 `main()` 函数末尾添加以下代码：

```mojo title="vector_addition.mojo"
        # 为输入向量创建 DeviceBuffer
        lhs_device_buffer = ctx.enqueue_create_buffer[float_dtype](vector_size)
        rhs_device_buffer = ctx.enqueue_create_buffer[float_dtype](vector_size)

        # 将输入向量从 HostBuffer 复制到 DeviceBuffer
        ctx.enqueue_copy(dst_buf=lhs_device_buffer, src_buf=lhs_host_buffer)
        ctx.enqueue_copy(dst_buf=rhs_device_buffer, src_buf=rhs_host_buffer)

        # 为结果向量创建 DeviceBuffer
        result_device_buffer = ctx.enqueue_create_buffer[float_dtype](
            vector_size
        )
```

[`DeviceBuffer`](/docs/std/gpu/host/device_context/DeviceBuffer/) 类型类似于 `HostBuffer` 类型，但表示与特定 `DeviceContext` 关联的设备内存块。具体来说，缓冲区位于设备的*全局内存*空间中，可由设备上执行的所有线程访问。与 `HostBuffer` 一样，`DeviceBuffer` 受 Mojo 标准所有权和生命周期机制的约束。它会持久存在直到程序中不再被引用或 `DeviceContext` 本身被销毁。

[`enqueue_create_buffer()`](/docs/std/gpu/host/device_context/DeviceContext/#enqueue_create_buffer) 方法接受数据类型作为编译时参数，缓冲区大小作为运行时参数，并返回一个 `DeviceBuffer`。该操作是异步的，但我们还不需要调用 `synchronize()` 方法，因为我们有更多操作要添加到队列中。

[`enqueue_copy()`](/docs/std/gpu/host/device_context/DeviceContext/#enqueue_copy) 方法被重载以支持从主机到设备、设备到主机，甚至对于有多个 GPU 的系统从设备到设备的复制。在此示例中，我们使用它将 `HostBuffer` 中的数据复制到 `DeviceBuffer`。

:::note

`DeviceBuffer` 和 `HostBuffer` 都包含 [`enqueue_copy_to()`](/docs/std/gpu/host/device_context/DeviceBuffer/#enqueue_copy_to) 和 [`enqueue_copy_from()`](/docs/std/gpu/host/device_context/DeviceBuffer/#enqueue_copy_from) 方法。这些只是调用其对应 `DeviceContext` 上 `enqueue_copy()` 方法的便捷方法。因此，我们可以用以下等效代码编写前面示例中的复制操作：

```mojo
    lhs_host_buffer.enqueue_copy_to(dst=lhs_device_buffer)
    rhs_host_buffer.enqueue_copy_to(dst=rhs_device_buffer)
```

:::

## 8. 创建 `TileTensor` 视图

在编写内核函数之前的最后一步是为每个向量创建 [`TileTensor`](/docs/layout/tile_tensor/TileTensor/) 视图。`TileTensor` 为多维数据提供了强大的抽象，可以精确控制内存组织。它支持各种内存布局（行主序、列主序、分块）、硬件特定优化和高效并行访问模式。我们在本教程中不需要所有这些功能，但在更复杂的内核中，它是操作数据的有用工具。尽管对于此示例并非严格必要，我们将使用 `TileTensor` 以便你可以熟悉它。

首先，在文件顶部添加以下导入：

```mojo title="vector_addition.mojo"
from layout import TileTensor, row_major
```

[`Layout`](/docs/layout/tile_layout/Layout/) 是使用形状和步幅信息的内存布局表示，它在逻辑坐标和线性内存索引之间映射。我们需要多次使用相同的 `Layout` 定义，所以在文件顶部的其他 `comptime` 声明之后添加以下 `comptime` 值：

```mojo title="vector_addition.mojo"
comptime layout = row_major[vector_size]()
```

最后，在 `main()` 函数末尾添加以下代码为每个向量创建 `TileTensor` 视图：

```mojo title="vector_addition.mojo"
        # 将 DeviceBuffer 包装在 TileTensor 中
        lhs_tensor = TileTensor(lhs_device_buffer, layout)
        rhs_tensor = TileTensor(rhs_device_buffer, layout)
        result_tensor = TileTensor(result_device_buffer, layout)
```

## 9. 定义向量加法内核函数

现在我们准备编写内核函数。首先，添加以下导入（注意我们从 `gpu` 的导入列表中添加了 `block_dim`）：

```mojo title="vector_addition.mojo"
from std.gpu import block_dim, block_idx, thread_idx
from std.math import ceildiv
```

然后，在 `main()` 函数之前将以下代码添加到 `vector_addition.mojo`：

```mojo title="vector_addition.mojo"
# 通过将向量大小除以块大小并向上取整来计算所需的线程块数。
comptime block_size = 256
comptime num_blocks = ceildiv(vector_size, block_size)


def vector_addition(
    lhs_tensor: TileTensor[float_dtype, type_of(layout), MutAnyOrigin],
    rhs_tensor: TileTensor[float_dtype, type_of(layout), MutAnyOrigin],
    out_tensor: TileTensor[float_dtype, type_of(layout), MutAnyOrigin],
):
    """在 GPU 上计算两个向量的逐元素和。"""

    # 计算线程要处理的向量元素的索引
    var tid = block_idx.x * block_dim.x + thread_idx.x

    # 不处理越界元素
    if tid < vector_size:
        out_tensor[tid] = lhs_tensor[tid] + rhs_tensor[tid]
```

我们的 `vector_addition()` 内核函数接受两个输入张量和一个输出张量作为参数。我们还需要知道向量的大小（我们用 `comptime` 值 `vector_size` 定义了），因为它可能不是块大小的倍数。实际上，在这个示例中，向量的大小是 1,000，这不是我们块大小 256 的倍数。因此，在将线程分配给从张量读取元素时，我们需要确保不会超出张量的边界。

内核函数的主体首先计算特定线程负责处理的张量元素的线性索引。`block_dim` 对象（我们添加到导入列表中）包含线程块的维度，作为 `x`、`y` 和 `z` 值。因为我们将使用一维线程块网格，我们只需要 `x` 维度。然后我们可以计算 `tid`，即线程在输出张量中的唯一"全局"索引，计算方式为 `block_dim.x * block_idx.x + thread_idx.x`。例如，第一个线程块中线程的 `tid` 值范围从 0 到 255，第二个线程块中线程的 `tid` 值范围从 256 到 511，依此类推。

:::note

为方便起见，[`gpu`](/docs/std/gpu/) 包包含一个 `global_idx` `comptime` 值，其中包含线程在线程块网格中的唯一"全局" `x`、`y` 和 `z` 索引。因此，对于我们的一维线程块一维网格，`global_idx.x` 等同于我们上面计算的 `tid` 值。但是，对于本教程，最好学习如何手动计算 `tid`，以便你了解网格和线程块维度是如何工作的。

:::

然后函数检查计算的 `tid` 是否小于输出张量的大小。如果是，线程从 `lhs_tensor` 和 `rhs_tensor` 张量读取相应元素，将它们相加，并将结果存储在 `out_tensor` 张量的相应元素中。

## 10. 调用内核函数并将输出复制回 CPU

最后一步是编译和调用内核函数，然后将输出复制回 CPU。在 `main()` 函数末尾添加以下代码：

```mojo title="vector_addition.mojo"
        # 编译并入队内核
        ctx.enqueue_function[vector_addition](
            lhs_tensor,
            rhs_tensor,
            result_tensor,
            grid_dim=num_blocks,
            block_dim=block_size,
        )

        # 为结果向量创建 HostBuffer
        result_host_buffer = ctx.enqueue_create_host_buffer[float_dtype](
            vector_size
        )

        # 将结果向量从 DeviceBuffer 复制到 HostBuffer
        ctx.enqueue_copy(
            dst_buf=result_host_buffer, src_buf=result_device_buffer
        )

        # 最后，同步 DeviceContext 以运行所有入队操作
        ctx.synchronize()

        print("结果向量:", result_host_buffer)
```
